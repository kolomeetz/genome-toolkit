# PDF Export Design — Doctor-Ready Reports

Issue: #12

## Approach: Hybrid (Markdown-to-PDF via backend)

### Why not frontend HTML-to-PDF?

- `html2pdf.js` / `jsPDF` produce mediocre typography, poor table handling, and inconsistent pagination
- Browser `window.print()` already exists for quick prints (`printPage()` in `export.ts`) — no need to duplicate it
- Clinical reports need reliable page breaks, headers/footers, and consistent rendering across environments

### Why backend Markdown-to-PDF?

- The codebase already generates clean Markdown for every view (`pgxToMarkdown`, `mentalHealthToMarkdown`, `riskLandscapeToMarkdown`, `addictionToMarkdown`, `checklistToMarkdown`)
- The user already has pandoc + Eisvogel LaTeX template with EB Garamond fonts for PDF generation
- Adding one backend endpoint (`POST /api/export/pdf`) that receives Markdown and returns a PDF is minimal work
- Produces print-quality typography with proper page breaks, headers, footers, page numbers

### Architecture

```
Frontend                          Backend
────────                          ───────
User clicks "Export PDF"
  → existing *ToMarkdown()
  → POST /api/export/pdf          → receive { markdown, report_type, metadata }
     { markdown, type, meta }     → inject YAML front matter (title, date, header)
                                  → pandoc --template eisvogel -o report.pdf
  ← binary PDF response           ← return PDF bytes
  → trigger browser download
```

New files:
- `backend/app/routes/export.py` — single route, `POST /api/export/pdf`
- `backend/app/services/pdf_renderer.py` — pandoc wrapper, template config
- `frontend/src/lib/export.ts` — add `exportPdf(markdown, type, metadata)` function

Dependencies: `pandoc` (already available), Eisvogel template (already installed in user's system). No new Python packages needed — uses `subprocess.run` to call pandoc.

## Report Types

### 1. PGx Card (1-2 pages)

**Purpose:** Hand to prescriber before medication changes. Most time-sensitive report.

**Sections:**
- Header: patient identifier (optional), date, "PHARMACOGENOMICS REPORT"
- Disclaimer box: "Not medical advice. Discuss with prescriber."
- Enzyme summary table: symbol, alleles, metabolizer status, guideline source
- Per-enzyme drug impact list: drug class, impact level (DANGER/WARN/ADJUST/OK), status text, affected drugs
- Danger notes called out in bordered boxes
- Substances section (if user opts in — checkbox in export dialog)
- Footer: "Generated by Genome Toolkit", date, page number

**Data source:** `pgxToMarkdown()` output from `PGxEnzymeSection[]`

**Fields per enzyme:**
- `enzyme.symbol`, `enzyme.alleles`, `enzyme.status`, `enzyme.geneType`, `enzyme.guideline`
- Per drug: `drugClass`, `impact`, `statusText`, `drugList`, `dangerNote`, `category`

### 2. Mental Health Summary (2-3 pages)

**Purpose:** Share with psychiatrist/therapist for context on genetic factors.

**Sections:**
- Header: date, "MENTAL HEALTH GENETIC PROFILE"
- Overview stats: actionable / monitor / optimal gene counts, total actions
- Per-pathway block:
  - Pathway name + status badge
  - Narrative body text
  - Gene table: symbol, variant, genotype, status, evidence tier, study count, description
  - Actions list per gene: type (CONSIDER/MONITOR/DISCUSS/TRY), title, description, done status
- GWAS findings section (if available): trait, significant hits, odds ratios
- Checklist of pending actions
- Footer

**Data source:** `mentalHealthToMarkdown()` output from `PathwaySection[]` + `Record<string, ActionData[]>`

### 3. Full Genome Report (5-10 pages, comprehensive)

**Purpose:** Complete reference document combining all views.

**Sections:**
1. Title page: "PERSONAL GENOME REPORT", date, disclaimer
2. Table of contents (auto-generated by pandoc)
3. Executive summary: key findings across all domains (1 paragraph each)
4. Mental Health Profile (from report type 2)
5. PGx / Drug Metabolism (from report type 1)
6. Mortality & Risk Landscape:
   - Risk table: rank, cause, % of deaths, status, genes
   - Expanded detail per actionable/monitor cause: narrative, genes, actions
   - Data source: `riskLandscapeToMarkdown()` from `MortalityCause[]`
7. Addiction & Reward Profile:
   - Pathway sections with gene tables
   - Substance-specific harm reduction cards
   - Data source: `addictionToMarkdown()` from `PathwaySection[]` + `SubstanceCard[]`
8. Action Checklist: consolidated from all views, grouped by status
9. Appendix: evidence tier definitions, methodology note, data sources

**Data source:** all `*ToMarkdown()` functions concatenated with section headers

## Visual Style

### Typography
- Body: EB Garamond (serif) — already configured in Eisvogel template
- Headers: EB Garamond Bold, uppercase with letter-spacing for section titles
- Data tables and gene/variant identifiers: monospace (matches app's `var(--font-mono)`)
- Status labels: small caps monospace

### Color (print-safe)
- Black text on white background — no dark theme
- Status indicators use grayscale-safe patterns:
  - DANGER/Actionable: solid black left border + bold text
  - WARN/Monitor: dashed left border
  - OK/Optimal: thin gray left border
  - Evidence tiers: E1-E5 shown as text labels, not color-coded
- Tables use alternating light gray row shading

### Layout
- A4 page size (European standard per user prefs)
- Margins: 25mm top/bottom, 20mm left/right
- Header: report type + date (right-aligned)
- Footer: "Genome Toolkit" + page number
- Gene tables: compact, no wrapping on symbol/variant/genotype columns
- Danger notes: bordered box with "WARNING" prefix in bold

### Pandoc/Eisvogel configuration (YAML front matter)
```yaml
titlepage: true
titlepage-color: "FFFFFF"
titlepage-text-color: "1a1a1a"
toc: true                    # only for Full Report
toc-depth: 2
header-left: "GENOME TOOLKIT"
header-right: "\\today"
footer-center: "\\thepage"
mainfont: "EB Garamond"
monofont: "JetBrains Mono"
fontsize: 11pt
geometry: "a4paper,margin=20mm"
colorlinks: true
```

## Export UI

Add to each view's toolbar (next to existing "Export" button):

```
[Export PDF ▾]
  ├─ PGx Card (1-2 pages)
  ├─ Mental Health Summary (2-3 pages)
  ├─ Full Genome Report
  └─ ☐ Include substances/harm reduction
```

The dropdown calls the appropriate `*ToMarkdown()` function, then `exportPdf()`.

For Full Report, the frontend concatenates all markdown outputs with `---` separators and `\newpage` LaTeX commands between sections.

---

## Implementation Plan

1. **Backend route** (`export.py`): accept POST with `{ markdown: string, metadata: object }`, write temp file, run pandoc, return PDF, clean up
2. **PDF renderer service** (`pdf_renderer.py`): build YAML front matter from metadata, write combined `.md` file, invoke pandoc with Eisvogel template
3. **Frontend `exportPdf()`** in `export.ts`: POST markdown to backend, receive blob, trigger download
4. **UI**: add PDF export dropdown to `FilterBar` / `Toolbar` components in each view
5. **Testing**: verify pandoc is available on startup, return helpful error if not

Estimated effort: ~4 hours for a working v1.
